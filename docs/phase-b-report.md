## 1. 全体像

### 1.1 何を作ったか（一言で）

**「StackChan の画面をタップして話しかけると、Hermes Agent が考えて、StackChan の声で返事をする」** を実現しました。

達成した体験:

```
あなた:「すたっくちゃん、こんにちは。元気ですか？」
   （タップ → 話す → 黙る。約5〜7秒後…）
StackChan:「こんにちは！元気だよ」（VOICEVOX の声で）
```

### 1.2 システム構成図

登場人物は3つ。**全部が自宅の中で完結**しています（クラウド音声サービスを使っていない）。

```
┌──────────────── 自宅 LAN (192.168.0.x) ────────────────┐
│                                                         │
│  [StackChan 実機]                [razer-server]          │
│   M5Stack CoreS3                 Ubuntu ノートPC          │
│   ・マイクで録音                                          │
│   ・スピーカーで再生    WiFi      ┌─────────────────┐    │
│   ・画面タップ検知   ◄────────►  │ gateway          │    │
│   ・VAD(無音検出)    WebSocket   │ (このリポジトリの │    │
│                      :8765      │  Python プログラム)│    │
│                                 │   │ ①音声→文字    │    │
│                                 │   │ (faster-      │    │
│                                 │   │  whisper)     │    │
│                                 │   ▼               │    │
│                                 │ ②文字→Hermes へ  │    │
│                                 │   │ HTTP :8642    │    │
│                                 │   ▼               │    │
│                                 │ [Hermes Agent]    │    │
│                                 │  「思考体」        │    │
│                                 │   │ 返答テキスト   │    │
│                                 │   ▼               │    │
│                                 │ ③文字→音声       │    │
│                                 │  (VOICEVOX :50021)│    │
│                                 └─────────────────┘    │
│                                                         │
└─────────────────────────────────────────────────────────┘
```

**ポイント**: StackChan 本体（ESP32 マイコン）は「耳と口と顔」だけを担当し、
「脳」は全部 razer-server 側にあります。マイコンは非力なので、重い処理
（音声認識・AI 思考・音声合成）は LAN 内の PC に任せる分業です。
これは本プロジェクトの核心コンセプト「**依代**（よりしろ）— 思考体 Hermes が
器 StackChan に宿る」の実装そのものです。

### 1.3 会話1往復で何が起きているか（データの旅）

```
 1. タップ          → StackChan が録音開始（LED 緑点灯）
 2. 話す            → 声を Opus 形式（音声の圧縮形式）に圧縮しながら
                      WebSocket で gateway へリアルタイム送信
 3. 黙る            → VAD が「1.2秒無音」を検知して録音を自動終了 ★Phase B 後半で実装
 4. STT             → faster-whisper が音声を文字に変換（約0.6秒）
 5. Hermes が思考    → 「お元気ですか」→「元気だよ！」（約3秒）
 6. TTS             → VOICEVOX が返答テキストを音声に合成（約2〜4秒）
 7. 再生            → Opus に圧縮して StackChan へ送信、スピーカーから再生
```

### 1.4 常駐サービス構成（systemd）

razer-server の電源を入れれば全部勝手に立ち上がる構成にしました:

| サービス名 | 役割 | ポート |
|---|---|---|
| `stackchan-gateway` | 本リポジトリの gateway | :8765 (WS) / :8766 (HTTP) / :8767 (MCP) |
| `voicevox` | 音声合成エンジン | :50021 |
| `hermes-gateway` | Hermes Agent（Discord + API） | :8642 |

## 2. 採用した技術の解説

それぞれ「何か → なぜ必要か → 喩え → 使った場面」の順で説明します。

### 2.1 WebSocket — 「繋ぎっぱなしの電話」

- **何か**: PC とデバイスの間で**接続を張りっぱなしにして双方向に**データを送り合う通信方式。
- **なぜ必要か**: 普通の Web 通信（HTTP)は「手紙」方式 — 毎回こちらから出さないと返事が来ません。
  でも StackChan には「gateway 側から好きなタイミングで命令を送る」「録音を流れ続けるまま受け取る」
  必要があるので、**電話のように繋ぎっぱなし**にできる WebSocket を使います。
- **使った場面**: StackChan ⇔ gateway 間の常時接続（ポート 8765）。録音音声・再生音声・
  「録音開始/停止」などの命令が全部この1本の電話の上を流れます。

### 2.2 Opus — 「音声専用の ZIP 圧縮」

- **何か**: 音声データの圧縮形式。生の音声（PCM）はサイズが大きいので、送る前に小さくします。
- **なぜ必要か**: 16kHz の生音声は1秒あたり 32KB。WiFi 上のマイコンには重いので、
  Opus で約 1/10 に圧縮してから送ります。電話や Discord の通話も内部では Opus です。
- **使った場面**: StackChan ⇔ gateway の音声は全部「60ミリ秒ぶんの Opus の小包（フレーム）」
  単位で流れます。「録音 498 フレーム = 約30秒」のように、フレーム数 × 60ms で長さが分かります。

### 2.3 faster-whisper — 「耳」（STT: 音声→文字）

- **何か**: OpenAI が公開した音声認識モデル Whisper の高速版。**自宅 PC の中だけで**動きます。
- **なぜ必要か**: Hermes は文字しか読めないので、声を文字に変換する係が必要です。
- **設計判断**: GPU のメモリ（VRAM 6GB）が貴重なので、**CPU で動かす設定**（int8 という軽量化方式）
  を選びました。それでも転写は約 0.6 秒と十分速い。
- **学んだこと**: 初回だけモデルの読み込みで数秒かかる（2回目からは速い）。
  これを「ウォームアップ」と呼びます。

### 2.4 VOICEVOX — 「口」（TTS: 文字→音声）

- **何か**: 無料で使える日本語音声合成ソフト。「ずんだもん」などで有名なやつです。
- **なぜ必要か**: Hermes の返答（文字）を声にする係。
- **裏話**: 過去の別プロジェクトの残骸としてゴミ箱（`~/trash/`)行きになっていた
  エンジン本体 2.1GB を `~/apps/voicevox/` に救出して再利用しました。

### 2.5 MCP (Model Context Protocol) — 「AI 用の共通リモコン規格」

- **何か**: AI エージェントが外部の道具（ツール）を呼び出すための共通規格。
  Claude も Hermes も、この規格を喋れるツール箱になら何でも繋げます。
- **なぜ必要か**: Hermes に「StackChan を操作する能力」を与えるため。
  gateway は MCP サーバーとして `say`（喋らせる）`move_head`（首を動かす）など
  30個のツールを Hermes に見せています。
- **使った場面**: Hermes → gateway（ポート 8767）。Discord で「StackChan で喋って」と
  頼むと、Hermes が MCP 経由で `say` ツールを呼びます。

### 2.6 VAD (Voice Activity Detection) — 「人が喋ってるか検知する耳のセンサー」

- **何か**: 音声区間検出。マイク入力から「いま人の声がある/ない」をリアルタイム判定する技術。
  ESP32 の音声前処理ライブラリ（AFE = Audio Front End、espressif 製）に内蔵されています。
- **なぜ必要か**: 「話し終わったのに録音が30秒続く」問題の解決（→ 章3の事件簿2）。
- **重要な設計**: VAD を「常時聞き耳」には**使っていません**。タップで明示的に始めた録音の
  「終わりの検出」にだけ使います。設計原則1（夫婦の会話に割り込まない）は守られています。

### 2.7 systemd — 「Linux の自動スタッフ管理者」

- **何か**: Linux でプログラムを「サービス」として常駐させる仕組み。
  電源投入で自動起動、落ちたら自動再起動してくれます。
- **なぜ必要か**: 開発中は手でプログラムを起動していましたが、それだと PC を再起動するたび
  に手作業が要ります。「StackChan は家電のように、何もしなくても動いている」状態にするため、
  3つのサービスを systemd 管理にしました。
- **便利技**: **drop-in** という仕組みで、元の設定ファイルを書き換えずに
  「追加の設定だけ別ファイルで上書き」できます（`サービス名.service.d/xxx.conf`）。
  本体ファイルを汚さないので、後から「何を変えたか」が一目瞭然です。

> **教訓**: 「Phase 4 自前実装: Opus 音声ストリーム」とロードマップに書いてあった作業は、
> 実は上流（fork 元）が既に作ってくれていた。**書く前に読む**ことで数日分の作業が消えました。

### 工程3: 足回り（B1〜B4）

| 作業 | 内容 |
|---|---|
| B1 | gateway に STT/TTS 用ライブラリ導入、VOICEVOX 救出・復旧 |
| B2 | TTS 単体テスト: `say` → VOICEVOX → 実機へ送信成功 |
| B4 | Hermes の API サーバー機能を有効化（外から HTTP で会話を注入できる口） |

### 工程4: 心臓部 — `/voice_turn` ブリッジ（B5、コミット `1fd2ec4`)

このフォークで唯一の本格的な新規コード `gateway/stackchan_mcp/hermes_bridge.py` を書きました。
やっていることは「電話の交換手」です:

```python
# 概念コード（実物は gateway/stackchan_mcp/hermes_bridge.py）
async def handle_voice_turn(録音データ):
    文字 = STT(録音データ)           # faster-whisper
    返答 = await Hermes に聞く(文字)  # HTTP POST :8642
    await 喋らせる(返答)             # VOICEVOX → 実機
```

面白い構成: firmware は録音が終わると「録音データをどこかの URL に POST する」機能を
持っていたので、その宛先を **gateway 自身**（`http://127.0.0.1:8766/voice_turn`）に
向けました。自分で受けて自分で処理する「ループバック」構成で、新しいサーバーを
立てずに済んでいます。

セキュリティも最初から入れました: 受け取るデータのサイズ上限、解凍爆弾
（小さい圧縮データが巨大に展開される攻撃）対策、認証なしのアクセス拒否。

### 工程5: E2E テスト — 実機なしで全体を通す（B6 シミュレート版）

実機を触れない時間帯でも検証を進めるため、「VOICEVOX で合成した音声を、実機の録音と
同じ形式で `/voice_turn` に投げ込む」テストスクリプトを作りました（`scratch/test_voice_turn.py`）。

```
合成音声「好きな食べ物はある?」→ STT → Hermes「ラーメンかな」→ TTS → 実機へ
```

> **教訓**: ハードウェアが絡む開発では「実機がなくても回せるテスト経路」を最初に作ると
> 開発速度が桁違いになります。

### 工程6: 常駐化（B7〜B8）

手動起動だった gateway を systemd サービスにし、Hermes の設定に MCP サーバーとして登録。
認証キー（`HERMES_API_KEY`）も設定しました。

### 事件簿1: StackChan が勝手に消えた（電源オフ問題）

**症状**: 夕方、ESP32 が忽然とオフラインに。USB 給電なのに電源が切れていた。

**原因**: 上流の firmware に「WebSocket が切断されたまま5分経つと電源を切る」という
省電力タイマー（`PowerSaveTimer`）が仕込まれていました。gateway を入れ替え作業で
5分以上止めたので、デバイスが「見捨てられた」と判断して自殺していたのです。

**修正**: タイマーの「電源オフ」だけ無効化（画面減光は維持）。1行の変更です。

> **教訓**: fork したコードには「前の持ち主の事情」が埋まっています。不可解な挙動は
> まずログとコードで根本原因を特定する。「電源ボタンを押せば直る」で済ませていたら
> 毎日再発していました。

### 事件簿2: 返事が遅い（10〜30秒）→ VAD 自動停止（コミット `682c38f`）

**症状**: 実機で会話は成立したが、話しかけてから返事まで 10〜30 秒かかる。

**調査**: gateway のログに処理時間の内訳（`timings_ms`）を出していたので見たところ:

```
録音: きっかり 498 フレーム = 30秒 ← 犯人はここ！
STT: 1.4〜2.3秒 / Hermes: 2.6〜3.4秒 / 合成: 2〜4秒 ← 処理は速い
```

処理が遅いのではなく、**話し終わった後も録音が30秒の上限まで続いていた**のです。

**おまけの発見**: 音声認識が「ん」「オーブン」のような崩壊した転写を返していたのも
同じ原因でした。30秒の録音のほとんどが無音とノイズで、whisper が混乱していた。
**1つの根本原因が2つの症状（遅い・聞き取れない）を生んでいた**典型例です。

**修正**: ESP32 側で VAD を監視し、「発話を検知した後、無音が1.2秒続いたら録音を
自動終了」するようにしました。安全装置として:

- 録音開始直後 0.8 秒は判定しない（起動音を自分のマイクで拾って誤検知するため）
- 発話を一度も検知できていない間は止めない（声が小さくて VAD が拾えない場合に、
  あなたの発話を切り捨てないため。その場合は従来の30秒上限と再タップ送信が保険）

**結果**: 録音が発話分+1.2秒で終わるようになり、転写も「お元気ですか」と正確に。
発話終了→返答開始が**約5〜7秒**まで短縮されました。

### 工程7: 認証の仕上げ（STACKCHAN_TOKEN）

最後に、それまで「誰でも接続できる」状態だった StackChan ⇔ gateway の WebSocket に
合言葉（Bearer トークン）を設定しました。firmware 側はビルド時に焼き込み
（`sdkconfig.defaults.local`、git 管理外）、gateway 側は `~/.yorishiro/secrets.env`。
誤ったトークンでの接続が HTTP 401 で拒否されることもテストで確認済みです。

## 4. 設計判断の記録

### 4.1 なぜ「完全ローカル」にしたか（判断1）

| 観点 | (a) 完全ローカル ✅ | (b) クラウド併用 |
|---|---|---|
| プライバシー | 家庭内の音声が家から出ない | 音声がクラウドに送られる |
| ランニングコスト | 0円 | API 従量課金 |
| 応答速度 | 5〜7秒（現状） | 1〜2秒も可能 |
| 災害・障害耐性 | ネット断でも音声系は生存 | ネット必須 |

速度ではクラウドに負けますが、**家庭に置く機械として音声を外に出さない**価値を取りました。
速度はこの後も文分割 TTS などで改善余地があります（章6）。

### 4.2 なぜ Hermes との接続を「MCP + API サーバー」の2本立てにしたか（判断3)

方向が2つあることに注意してください:

- **Hermes → StackChan**（Hermes が能動的に器を操作）: MCP。Hermes は MCP クライアント
  機能を内蔵していたので、gateway を登録するだけで30個のツールが使えるようになった。
- **StackChan → Hermes**（あなたの発話を Hermes に届ける）: Hermes 内蔵の API サーバー。
  MCP は「AI が道具を使う」ための規格で、「外から AI に話しかける」用途には向かないため。

> **教訓**: 「AとBを繋ぐ」と一括りにせず、**矢印の向きごとに最適な手段を選ぶ**。

### 4.3 録音の止めどき — 3つの選択肢

| 方式 | 利点 | 欠点 | 採用 |
|---|---|---|---|
| 再タップで停止 | 確実・実装済みだった | 毎回2回タップは面倒 | 保険として維持 |
| 時間で打ち切り (30秒) | 実装が単純 | 短い発話で29秒無駄 | 上限の保険として維持 |
| VAD 自動停止 | 自然な UX | 声が小さいと検知漏れ | **主経路として採用** |

3つを**重ねて**実装しているのがポイントです。VAD が効けば最速、効かなくても
再タップか30秒で必ず終わる。「賢い機能が失敗したときに、素朴な機能が受け止める」
多層構造にしました。

### 4.4 やらなかったこと（意図的に）

- **VAD 常時起動（ウェイクワード的な使い方）**: 設計原則1に反するので不採用。
- **gateway の独自プロトコル追加**: firmware の既存機能（録音 POST、Opus 再生）に
  乗っかることで、fork 独自コードを `hermes_bridge.py` 1ファイルに抑えました。
  上流の更新を取り込みやすくするためです。
