## 1. 全体像

### 1.1 何を作ったか（一言で）

Phase B で「会話できる」ようになった StackChan を、**「速く答える・家電を操る」** に進化させました。

達成した体験:

```
あなた:「こんにちは」
StackChan:（約1秒で）「こんにちは！」       ← ローカル LLM 直行（C3）

あなた:「リビングの電気をつけて」
StackChan: 部屋の電気が実際に点く + 「つけたよ」  ← SwitchBot 連携（C2）
```

そして1つ、**やってみて初めて分かった失敗**もあります。CoreS3 に内蔵されている
近接センサーで「手をかざすと反応する」を作ろうとしたら、**本体の前面カバーが
センサーの光路を塞いでいて物理的に不可能**でした（C1、章3の事件簿2）。
これは外付けセンサー（ToF Unit）での再挑戦に切り替えています。

### 1.2 システム構成図（Phase C 後）

Phase B との違いは ★ 印の3つです。

### 1.3 会話1ターンの流れ（C3 のルーティング適用後）

Phase B では全部の発話が Hermes に行っていましたが、いまは**受付係**が入り口で振り分けます:

```
画面タップ → 録音（VAD 自動停止）→ faster-whisper で文字に
  → decide_route(文字)                ← ★C3 で追加した受付係
      ├─ 短い挨拶・雑談       → ローカル LFM2.5（約0.5秒）
      └─ 家電操作・調べ物・長文 → Hermes（約3秒、ツールが使える）
  → VOICEVOX で音声合成 → StackChan のスピーカーから再生
```

**ポイント**: 「どっちの脳に聞くか」を決めるのは AI ではなく、**ただのルール
（文字数とキーワード）**です。なぜ AI に判定させなかったかは章4で説明します。

## 2. 採用した技術の解説

それぞれ「何か → なぜ必要か → 喩え → 使った場面」の順で説明します。

### 2.1 SwitchBot Cloud API — 「家電メーカーの電話受付」

- **何か**: SwitchBot 社が公開している Web API。インターネット経由で
  「この部屋の電気をつけて」とリクエストすると、自宅の Hub Mini（赤外線リモコンの親機）
  が代わりに赤外線を飛ばしてくれます。
- **なぜ必要か**: StackChan（正確には Hermes）に家電を操作する手段を与えるため。
  Hub Mini はリビングと寝室に既設だったので、API の鍵を発行するだけで使えました。
- **喩え**: 家電メーカーの**電話受付**。「○○号室の照明お願いします」と正しい身分証付きで
  電話すると、現地のスタッフ（Hub Mini）が動いてくれる。
- **使った場面**: gateway に `switchbot.py` を新規実装し、MCP ツール3つ
  （一覧 / 状態取得 / 操作）として Hermes に公開（コミット `f45d345`）。

### 2.2 HMAC-SHA256 署名 — 「改ざんできない封蝋」

- **何か**: 秘密鍵を使ってメッセージから「改ざん検知コード」を作る暗号技術。
  SwitchBot API はリクエストごとに `トークン + 時刻 + 使い捨て乱数` をこの方式で
  署名させます。
- **なぜ必要か**: 単なる合言葉（Bearer トークン）方式だと、通信を盗み見られた場合に
  同じリクエストを**再送（リプレイ攻撃）**されます。時刻と使い捨て乱数（nonce）を
  署名に混ぜることで、盗んだリクエストは二度と使えなくなります。
- **喩え**: 手紙の**封蝋（ふうろう）**。中身と差出時刻を混ぜ込んだ印を秘密の印鑑で
  押すので、本人以外は作れないし、古い手紙の使い回しもバレる。
- **使った場面**: `switchbot.py` の `build_auth_headers()`:

```python
# gateway/stackchan_mcp/switchbot.py より（要点のみ）
string_to_sign = f"{token}{t}{nonce}".encode("utf-8")   # トークン+時刻+乱数
sign = base64.b64encode(
    hmac.new(secret.encode(), msg=string_to_sign,
             digestmod=hashlib.sha256).digest()
).decode()
```

### 2.3 Ollama + LFM2.5-1.2B-JP — 「家に住み込みの速答係」

- **何か**: Ollama はローカル PC で LLM を動かすためのサーバー。LFM2.5-1.2B-JP は
  LiquidAI 製の**日本語特化・超小型（1.2B パラメータ、量子化後 0.7GB）**モデルです。
- **なぜ必要か**: Hermes は外部の大型モデルを使うため、どんな短い挨拶にも 2.6〜3.4 秒
  かかります。「こんにちは」に 3 秒は長い。**挨拶程度なら家に住み込みの小型モデルで
  十分**なので、短文だけ任せて約 0.5 秒に短縮しました。
- **喩え**: 会社の**電話の一次受付**。「営業時間は?」程度は受付がその場で即答し、
  難しい相談だけ専門部署（Hermes）に回す。
- **使った場面**: 実測ベンチで gemma3:4b（1.45〜2.03 秒）より速い LFM2.5（0.48〜0.50 秒）
  を採用。`keep_alive=30m` でモデルを GPU メモリに常駐させ、コールドスタート
  （約 3.4 秒）は 30 分に1回しか払わない設計です。

### 2.4 ルールベース・ルーティング — 「AI を使わない受付係」

- **何か**: 発話をローカル LLM と Hermes のどちらに送るかを、**文字数とキーワードだけ**で
  決める仕組み（`local_llm.py` の `decide_route()`）。
- **なぜ必要か**: 振り分け自体に LLM を使うと、振り分けだけで数百ミリ秒〜数秒かかり
  本末転倒です。ルールなら 0 秒で、しかも挙動が 100% 予測できます。
- **喩え**: 病院の**問診票によるトリアージ**。記入内容のチェック項目だけで「待合室へ/
  すぐ診察室へ」を機械的に判断する。判断者が疲れたり気分で変わったりしない。
- **使った場面**: 判定ロジックの全文はこれだけです:

```python
# gateway/stackchan_mcp/local_llm.py の decide_route()（要点のみ）
normalized = unicodedata.normalize("NFKC", text).strip()
if not normalized:
    return ROUTE_HERMES                       # 空 → 念のため Hermes
if any(marker in normalized for marker in HERMES_MARKERS):
    return ROUTE_HERMES                       # 「調べて」「つけて」等 → Hermes
if len(normalized) <= 30:
    return ROUTE_LOCAL                        # 30文字以下 → ローカル
return ROUTE_HERMES                           # それ以外 → Hermes
```

  入出力だけで決まる**純関数**（外部に一切アクセスしない関数）なので、
  単体テストが書きやすいのも利点です（テスト 332 行で全パターン検証済み）。

### 2.5 opt-in 設計 — 「スイッチを入れた人にだけ現れる新機能」

- **何か**: 環境変数（`STACKCHAN_LOCAL_LLM_MODEL` など）を設定したときだけ新機能が
  有効になり、未設定なら**従来と完全に同じ動き**をする実装方針。
- **なぜ必要か**: 新機能にバグがあっても「設定を消すだけ」で Phase B の安定状態に
  戻れます。ロールバック手順が要らない＝深夜に壊れても怖くない。
- **喩え**: 家の**ブレーカーが別系統**になっている増築部分。新しい部屋が漏電しても
  ブレーカーを落とせば母屋は無事。
- **使った場面**: C3 のローカル LLM（環境変数 + systemd drop-in）、C2 の SwitchBot
  （トークン未設定ならツールがエラーを返すだけで gateway 本体は無傷）。

### 2.6 LTR-553ALS-WA — 「スマホと同じ近接センサー」（C1 で格闘した相手）

- **何か**: CoreS3 の前面に内蔵されている照度（明るさ）+ 近接センサー。
  赤外線 LED を発射し、**反射して返ってくる量**で「目の前に物があるか」を判定します。
- **なぜ必要か**: 「近づくと StackChan が目線を向ける」反射（設計原則2: 低レベル反射は
  ファーム自律）の入力にする予定でした。
- **喩え**: スマホの「**耳に当てると画面が消える**」あの仕組みと同一原理・同類の部品です。
- **使った場面**: firmware にドライバを自作（コミット `c306f8a`、284 行）。
  チップとの会話には成功しましたが、**前面カバーに赤外線の通り道がなく**断念
  （詳細は章3の事件簿2）。ドライバと診断機能は将来のために残してあります。

### 2.7 I2C（内部バスと外部バス）— 「機器内の社内連絡網」

- **何か**: マイコンと周辺チップ（センサー等）をつなぐ2本線の通信規格。各チップには
  住所（アドレス）があり、LTR-553 は `0x23` 番地に住んでいます。
- **なぜ必要か**: 今回の教訓の核心はここでした。CoreS3 には I2C バスが**2系統**あります:
  - **内部バス**: 基板上のチップ用（LTR-553 はこちら）
  - **外部バス（Grove Port A）**: 後付けユニット用
- **喩え**: 会社の**内線電話網**。本社ビル内の内線（内部バス）と、別館への専用線
  （外部バス）は別の網。別館の電話帳（`i2c_scan` ツール）をいくら調べても
  本社の部署は載っていません。
- **学んだこと**: gateway の `i2c_scan` ツールは外部バスしか見ないため、最初は
  「近接センサーは無い」と誤って結論しかけました。**内蔵チップは回路図や公式 docs で
  確認する**のが正解です（ユーザーの指摘で発覚）。

## 3. 工程の流れ（時系列）

### 事件簿1: 朝起きたら Hermes が StackChan を見失っていた（C-fix）

**症状**: Phase B の仕上げで認証（STACKCHAN_TOKEN）を有効化した直後から、
Hermes が gateway の MCP ツールを一切呼べなくなっていた。

**調査**: Hermes のログに決定的な手がかりがありました:

```
05:02 リトライ間隔 2s → 4s → 8s → 16s → 32s → giving up
```

この**倍々に伸びるリトライ間隔（指数バックオフ）**は「クライアントが自動再接続に
失敗し続けている」典型的なサインです。gateway 側は 401（認証エラー）を返していました。

**原因**: gateway の入り口に鍵をかけたのに、**その入り口を使う側（Hermes）に
鍵を渡し忘れた**。ESP32 には渡したのに、もう1人の利用者を忘れていたのです。

**修正**: Hermes の設定に認証ヘッダーを1行追加（`Authorization: Bearer ${STACKCHAN_TOKEN}`）。
Hermes は設定値の `${変数名}` を環境変数から展開できるので、トークンの実物は
設定ファイルではなく `~/.hermes/.env` に置けました。

> **教訓**: 認証を導入するときは「**この口を使うクライアントは全部で何人いるか**」を
> 先に数える。1人でも漏れると、その人は静かに締め出される。

### 工程1: C2 — SwitchBot 家電操作（コミット `f45d345`）

1. **調査**: 当初案は Home Assistant（家電統合ソフト）経由でしたが、調べると
   HA はこの家のどこにも動いていない。ゼロから立てるより **SwitchBot の公式 API に
   直結する薄いクライアントを自作**する方が早いと判断（→ 章4.1）
2. **準備**: SwitchBot アプリで開発者トークンを発行 → `~/.yorishiro/secrets.env` へ
3. **実装**: `switchbot.py`（209 行）+ テスト（378 行）。MCP ツール3つを既存の
   :8767 に**同居**させたので、新サービス・新ポート・Hermes 側の再登録はゼロ
4. **設計の小技**: SwitchBot ツールは ESP32 と無関係なので、サーボが暴れないように
   命令を1つずつ直列実行する仕組み（single-flight queue）を**バイパス**。
   家電操作が首振りの順番待ちをする意味はないからです

**E2E 確認**: 音声で「SwitchBot に登録されているデバイスを教えて」→ Hermes がツールを
呼んで実デバイス一覧を音声で報告 → 「リビングの電気をつけて」→ **実際に電気が点いた** ✅

### 工程2: C3 — 応答ルーティング層（コミット `59a7774`）

1. **モデル選定は実測で**: 候補2つを同じ質問で計測 — LFM2.5-1.2B-JP（0.48〜0.50 秒）
   vs gemma3:4b（1.45〜2.03 秒）。カタログ値ではなく**この PC での実測**で決定
2. **実装**: `local_llm.py`（205 行）+ テスト（332 行）。Phase B の心臓部
   `hermes_bridge.py` への変更は「受付係を1人挟む」50 行だけ
3. **フォールバック**: ローカル LLM が失敗（タイムアウト・Ollama 停止等）したら
   **黙って Hermes に回す**。会話が死ぬことはない
4. **小型モデルの弱点ケア**: 1.2B モデルは「今日が何日か」を知らないので、
   システムプロンプトに現在日付を注入（「今日は何曜日?」は短文なので必ずローカルに
   来てしまうため）

**E2E 前に見つけた穴**: 「電気をつけて」は7文字 — 文字数ルールだとローカル行きですが、
**ローカルモデルはツールを呼べない**ので「つけたよ〜」と言うだけで何も起きません。
家電操作の語（つけて/消して/電気/エアコン…）を Hermes 行きキーワードに追加して解決。

> **教訓**: 複数の機能（C2 の家電 × C3 の振り分け）を同時に作るときは、
> **機能どうしの交差点**にバグが湧く。単体テストが通っても、組み合わせの
> シナリオ（「短い家電命令」）を歩いてみるまで安心しない。

### 事件簿2: 内蔵近接センサーは「壁の向こう」だった（C1）

**経緯**: `i2c_scan`（外部バス）が空 → 「センサー無し、外付け購入かな」と結論しかけた
ところで、**ユーザーの指摘**により CoreS3 内蔵の LTR-553 を発見（公式 docs に記載、
firmware が単にドライバを持っていなかっただけ）。内部バス 0x23 で応答も確認。

**実装**: 既存の背面タッチ（Si12T）と同じ「100ms ごとのポーリング」パターンを踏襲して
ドライバ + 手かざしリフレックスを実装、ビルドして実機に書き込み。

**症状**: チップは生きている（部品 ID 0x92 を正しく返す）のに、手を 1cm まで
近づけても近接の生値（ps_raw）が**ピクリとも動かない**。

**調査 — 「設定の問題か、物理の問題か」の切り分け**: 3点セットで再書き込みしました:

1. 感度を仕様の上限へ（ゲイン x64・赤外線 LED 15 パルス・電流 100mA）
2. センサーの飽和フラグ（強すぎる光で振り切れた印）を可視化
3. 2 秒ごとに生値をログへ垂れ流す

結果: **生値は常に ~380 で固定、手に無反応**。「ベースラインは出る = IC は動いている」
「外界に無反応 = 光の通り道がない」で、**前面カバー（シェル）に LTR-553 用の開口が
なく、赤外線が物理的に外へ出られない**と確定しました。~380 という値は、カバーの
内側で反射した自分の赤外線（クロストーク）だけを見ている数字です。

**おまけの発見**: 初版の検知しきい値 200 はベースライン 380 より**低かった**ため、
起動するたびに「誰もいないのに首が上を向く」誤発動が起きていました。
リフレックスを無効化 + しきい値 700 に修正して再書き込み済み。

**結末**: 仮にカバーに穴を開けても有効距離は ~10cm（手かざし専用）。本来の目標
「近づくと向く」（1〜2m）には **M5Stack ToF Unit（VL53L0X、レーザー測距 2m、
Grove 接続、約¥1,000）**が適切と判断し、購入判断待ちに。ドライバ・ポーリング・
リフレックスの構造はそのまま VL53L0X 用に流用できます。

> **教訓その1**: 「動かない」の切り分けは**生値の連続ログ**が最強。
> 「ベースラインは出るか（= IC は生きてるか）」「刺激に反応するか（= 経路はあるか）」
> の2つが見えれば、書き込み1往復で結論が出る。最初からデバッグの可視性を仕込む価値がある。
>
> **教訓その2**: データシート上「存在する」部品が**使える**とは限らない。
> 筐体・光路・配線という物理レイヤーは、コードをいくら読んでも分からない。

## 4. 設計判断の記録

### 4.1 なぜ Home Assistant を立てず SwitchBot API 直結にしたか（C2）

| 観点 | (a) HA を新規構築 | (b) SwitchBot API 直結 ✅ |
|---|---|---|
| 追加で動かすもの | HA 本体（常駐・要保守） | なし（gateway に 209 行追加のみ） |
| 対応機器の幅 | 広い（メーカー横断） | SwitchBot のみ |
| 今すぐ動くか | 構築・学習コストが先 | トークン発行だけで即日 |
| 将来の拡張 | — | Hermes に HA 用ツールが眠っており、将来 HA を立てれば乗り換え可 |

今の家の機器は**全部 SwitchBot Hub 経由**なので、HA の汎用性は現時点では過剰装備。
「将来 HA が欲しくなったら乗り換えられる」逃げ道を確認した上で、最小の道を選びました。

### 4.2 なぜ振り分けを LLM にやらせなかったか（C3）

| 観点 | ルールベース ✅ | LLM 分類器 |
|---|---|---|
| 振り分け自体の所要時間 | 0 秒 | 数百ms〜数秒（本末転倒） |
| 挙動の予測可能性 | 100%（テストで全列挙可） | たまに気まぐれな誤分類 |
| 賢さ | 杓子定規 | 文脈を読める |

「速くするための仕組み」が自分で時間を食ったら意味がないので、ルール一択でした。
杓子定規さの副作用（例: 長めの雑談も Hermes に行って 3 秒かかる）は許容範囲。
**間違えても「ちょっと遅い」だけで「壊れる」ことはない**方向に倒しています
（逆方向の間違い = ツールが必要な発話がローカルに行く、だけは
キーワードで防いでいる。これが章3工程2の「穴」の話です）。

### 4.3 多層フォールバック — Phase B からの一貫した型

Phase B の録音停止（VAD → 再タップ → 30 秒上限）と同じ「賢い機能 + 素朴な保険」を
今回も踏襲しています:

```
ローカル LLM（賢く速い）
  → 失敗したら Hermes（確実）
     → それも失敗したら定型の謝罪音声（Phase B から）
```

### 4.4 やらなかったこと（意図的に）

- **C1 のためにシェルへ穴あけ加工**: 開口しても手かざし（~10cm）専用にしかならず、
  本来の目標（1〜2m）に届かない。器物を不可逆に加工してリターンが小さい
- **SwitchBot のローカル API（Matter 等）**: Hub Mini はクラウド API が公式ルート。
  応答 5〜6 秒は許容し、確実に動く道を採用
- **ローカル LLM へのツール呼び出し付与**: 1.2B モデルにツールを持たせるのは精度的に
  危険（家電の誤操作は実害が出る）。ツールが要る発話は全部 Hermes へ
