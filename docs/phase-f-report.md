## 1. 全体像

### 1.1 何を作ったか（一言で）

これまでの Phase は「StackChan に新しい能力を足す」話でした（声・家電・自発発話）。
Phase F はそれと毛色が違って、**毎日触るときの操作性を整える**フェーズです。一言でいうと:

> **スマホのブラウザから StackChan を整えられるようにし、顔を見れば「今どの状態か」が分かるようにした。**

達成した体験:

```
・スマホのダッシュボードで音量・マイク感度・首の角度をその場で調整（再ビルド不要）
・話しかけると顔に「きいてるよ」→「考え中」→（検索なら「調べ中」）と出る
・首をいじっても 60 秒で正面・無表情・消灯に自動で戻る
・うるさかった「30分ごとの顔振り」をトグルで OFF にできる
```

### 1.2 なぜ Phase F は「4段」に枝分かれしたのか

このフェーズは一直線ではなく、**実機で触る → 不満が出る → 直す**を繰り返して 4 段に膨らみました。
これ自体が学びなので、最初に流れを示します。

| 段 | 何をやったか | きっかけ |
|---|---|---|
| **F**（6/13） | ダッシュボード操作カード・顔ステータス・仕草OFF・ウェイクワード初版 | 計画どおりの本体 |
| **F-2**（6/13） | 字幕・Hermes 判別（画面 "H" + 青LED）・マイクメーター・タブUI・日付バグ修正 | F を実機で見て「もっと見たい情報」が出た |
| **F-3**（6/13） | マイク感度のライブ調整・会話ログ・タブ順序入替 | 「ウェイクワードが鳴らない」の真因対策を兼ねて |
| **フォロー**（6/14） | ②ウェイクワードの最終診断とクローズ・①顔ステータスの遅延修正・③④首角度のライブ調整 | 実機で残った 4 つの不満を潰した |

> **教訓**: 「操作性」の機能は、仕様書から生まれるより**実機を触った瞬間の違和感**から生まれる。
> だからこの手の改善は、1回で完璧を狙わず「触る→直す」を回せる体制（＝再ビルド不要のライブ調整）に
> 寄せていくほど速くなる。F の後半が全部「ライブ調整化」に向かったのは偶然ではない。

### 1.3 システム構成図（Phase F で増えた「3層」）

Phase F の主役は、スマホのブラウザから firmware まで指示が届く**3層の経路**です。

```
[スマホ / PC のブラウザ（自宅 LAN or Tailscale）]
    │  HTTP GET/POST  /control/volume, /control/head, /control/neutral_pose ...
    ▼
[razer-dashboard  status_api.py  :8080]          ← 受付。トークンはここに隠す（★git管理外）
    │  プロキシ（Authorization: Bearer <token> と Host を付け直して 127.0.0.1 へ転送）
    ▼
[gateway  http_server.py  :8767  /control/*]      ← 本体。トークンを照合して通す/弾く
    │  in-process（同じプロセスの中に control.py / heartbeat / hermes_bridge）
    ├─ control.py … 音量/ミュート/マイク感度の state、listen 発火、顔ステータス送出
    ├─ hermes_bridge.py … 会話中の「きいてるよ→考え中→調べ中→消去」を駆動
    ▼  WebSocket MCP  :8765
[StackChan firmware  stackchan.cc]
    ├─ self.display.set_status_text / set_subtitle / set_route_badge（新規：avatar 前面のラベル）
    ├─ self.robot.set_neutral_pose（新規：首の中立角を NVS 保存）
    ├─ self.audio_speaker.set_mic_gain（新規：マイク感度を NVS 保存）
    └─ カスタムウェイクワード（MultiNet, assets パーティション）→ 検知でタップと同じ listen 開始
```

ポイントは「ブラウザは gateway を**直接叩けない**」こと。なぜそうしたかは 2.1 で説明します。

## 2. 採用した技術の解説

### 2.1 3層プロキシ + Bearer トークン — 「受付を通さないと奥に入れない」

- **何か（一言）**: ブラウザ → `status_api.py`（受付）→ `gateway`（本体）と、間に1枚かます構成。
  本体は「`Authorization: Bearer <合言葉>` が付いた、127.0.0.1（自分自身）からの通信」しか受けない。
- **なぜ必要か**: gateway をブラウザから直接触れるようにすると、合言葉（トークン）を**ブラウザに渡す**
  必要が出る。ブラウザに渡した秘密は、ページのソースを見れば誰でも読めてしまう。だから合言葉は
  サーバー側（受付）に隠し、ブラウザは受付に頼むだけにした。
- **喩え**: 会員制の店。客（ブラウザ）は会員証（トークン）を持たされず、受付（status_api）に
  「3番テーブルに水を」と頼む。受付が会員証を見せて厨房（gateway）に通す。客は会員証を一度も持たない。
- **使った場面**: `http_server.py` の認証ガード。`/control/*`・`/mcp`・`/status` だけ合言葉を要求する:

```python
# http_server.py（要点）
token_protected = path in {"/mcp", "/status"} or path.startswith("/control")
if self._token and token_protected:
    if request.headers.get("authorization") != f"Bearer {self._token}":
        return 401  # 合言葉が違えば門前払い
```

ダッシュボードの実体ファイル（`status_api.py` / `dashboard.html`）は **git 管理外**の `~/razer-dashboard/` に
置いてあります。理由は 4.3 で。

### 2.2 LVGL の重なり順と再描画 — 「黒板に書いても、前に出してめくらないと見えない」

顔ステータス（「きいてるよ」等）の表示で、Phase F は LVGL の落とし穴を2つ踏みました。
先に道具の説明をします。

- **何か（一言）**: **LVGL** は ESP32 の画面に図形や文字を描くライブラリ。画面上の部品（ラベル・画像）は
  紙芝居のように**重なり順**を持ち、後ろの部品は前の部品に隠れる。
- **なぜ必要か（で、なぜ罠か）**: StackChan の顔（avatar 画像）は、瞬き・口パクのたびに「最前面に出す」
  処理が走る。素直にステータスのラベルを置くと、**瞬きのたびに顔がラベルを覆い隠す**。
- **喩え**: 黒板に伝言を書いても、その前に大きなポスター（顔）を貼られたら読めない。
  だから「ラベルを書いたら、顔より前にめくり出す」をセットでやる必要がある。
- **使った場面（罠1：隠れる）**: 顔を前面化した**直後に**ステータス類を再度前面化するヘルパに集約した:

```cpp
// stackchan.cc PromoteOverlaysLocked() — 顔を前に出した後に必ず呼ぶ
void PromoteOverlaysLocked() {
    if (status_label_  && !hidden) lv_obj_move_foreground(status_label_);   // 「きいてるよ」
    if (subtitle_label_&& !hidden) lv_obj_move_foreground(subtitle_label_); // 字幕
    if (route_badge_   && !hidden) lv_obj_move_foreground(route_badge_);    // "H" バッジ
}
```

- **使った場面（罠2：更新したのに描かれない）**: CoreS3 は省電力のため、何も起きないと画面の
  再描画タイマーが自分で止まる。ラベルの文字を変えても**次の再描画まで反映されない**ので、
  3つの setter すべてで「即フラッシュ」を強制した:

```cpp
// stackchan.cc SetStatusText()（要点）— 文字を変えたら自分で描かせる
lv_label_set_text(status_label_, safe);
lv_obj_update_layout(status_label_);                 // レイアウト再計算
lv_obj_invalidate(status_label_);                    // 「ここ描き直して」と印
lv_refr_now(lv_obj_get_display(status_label_));      // 今すぐ画面に出す
```

> **教訓**: 「値を変えたのに画面に出ない」は、たいてい**変える処理**ではなく**描き直す処理**が
> 抜けている。GUI は「データを変える」と「再描画する」が別工程だと意識すると、この種のバグが減る。

### 2.3 NVS ランタイム設定 — 「電源を切っても消えない手帳」

Phase F で増えた調整項目（首の中立角・マイク感度）は、すべて **NVS** に保存して
**再ビルドなしで変えられる**ようにしました。

- **何か（一言）**: **NVS（Non-Volatile Storage）** は ESP32 のキーバリュー保存領域。
  再起動しても、ファームを焼き直しても消えない。
- **なぜ必要か**: 「首の角度を 45° から 38° に」のような**見た目の好み**を、毎回 25 分の再ビルドで
  合わせるのは時間の無駄。値を NVS に置けば、MCP ツール1回で変えて即保存できる。
- **喩え**: ソースコードに書く設定＝「印刷された取扱説明書」（変えるには刷り直し）。
  NVS＝「手元の手帳」（その場で書き換えられて、電源を切っても残る）。
- **使った場面**: この発想は Phase C の `set_proximity_config`（近接センサーの閾値）で確立した型で、
  Phase F は同じ型を2か所に再利用した:

```cpp
// stackchan.cc set_neutral_pose（要点）— 首の中立角を NVS namespace "stackchan_pose" に
Settings settings("stackchan_pose", true);
settings.SetInt("yaw", yaw);  settings.SetInt("pitch", pitch);
neutral_yaw_ = yaw;  neutral_pitch_ = pitch;     // メモリ上の中立も更新
WriteHeadAngles(neutral_yaw_, neutral_pitch_);   // その場で動かして見せる
```

```cpp
// mcp_server.cc set_mic_gain（要点）— マイク感度を NVS namespace "audio" に
gain = ((gain + 1) / 3) * 3;        // 3dB ステップに丸める
codec->SetInputGain((float)gain);   // ES7210 ADC に即適用
Settings("audio", true).SetInt("input_gain", gain);  // 次回起動でも復元（既定 30dB）
```

中立角については「中立を参照する場所が3つ（起動時・近接やタッチからの復帰・アイドル復帰）に
散らばっていた」のを、`neutral_yaw_`/`neutral_pitch_` という**1つの出所**に束ねたのが要点です（3.x で詳述）。

### 2.4 MultiNet カスタムウェイクワード — 「中国語の発音記号で日本語を当て書き」

- **何か（一言）**: **MultiNet** は Espressif（ESP32 のメーカー）の音声コマンド認識モデル。
  中国語ピンイン（発音のローマ字表記）でコマンド語を登録できる。
- **なぜ必要か（で、なぜ難しいか）**: 「スタックちゃん」という**日本語**のウェイクワードを鳴らしたい。
  だが専用学習済みモデル（WakeNet）には日本語語が無い。そこで MultiNet に
  「スタックちゃん」を中国語ピンイン `su ta ke qiang`（スー・ター・カ・チアン）で**当て書き**した。
- **喩え**: 英語しか書けないペンで「ありがとう」を `arigato` と書くようなもの。読めなくはないが、
  元の音とはズレる。そのズレが認識の壁になった（結末は事件簿4）。
- **使った場面**: `config.json` の `sdkconfig_append`（ビルド設定の上書き）に投入:

```jsonc
"CONFIG_USE_AFE_WAKE_WORD=n",            // 専用ウェイクワード検出は使わない
"CONFIG_USE_CUSTOM_WAKE_WORD=y",         // カスタム（MultiNet コマンド）を使う
"CONFIG_CUSTOM_WAKE_WORD=\"su ta ke qiang\"",   // ピンイン当て書き
"CONFIG_CUSTOM_WAKE_WORD_DISPLAY=\"スタックちゃん\"",
"CONFIG_CUSTOM_WAKE_WORD_THRESHOLD=20",  // 小さいほど敏感
"CONFIG_SR_MN_CN_MULTINET7_QUANT=y"      // MultiNet7（量子化版）
```

なぜ `sdkconfig` を直接いじらず `config.json` に書くのかは、ビルド手順が `idf.py set-target` で
sdkconfig を**作り直す**ため。直接編集は毎回消える。config.json の append は作り直しに耐える唯一の置き場です。

### 2.5 RMS / dBFS — 「声の大きさを数値で測る物差し」

事件簿4（ウェイクワード診断）の主役になった単位です。

- **何か（一言）**: **RMS** は音の波形の「実効的な大きさ」（二乗平均平方根）。**dBFS** は
  「デジタルで出せる最大音量を 0 とした相対値」。マイナスが大きいほど小さい音。
- **なぜ必要か**: 「マイクが本当に音を拾っているか」を、感覚（聞こえる/聞こえない）ではなく
  **数値**で確かめるため。16bit 音声の最大は 32767。RMS が 1000 くらいなら約 -29dBFS で
  音声認識に十分、110 くらいだと約 -49dBFS で痩せすぎ。
- **喩え**: 体重計。「太った気がする」ではなく数字で見れば、減らしすぎ（やつれ）も一目で分かる。
  実際このおかげで「マイク感度を下げすぎていた」過剰補正に気づけた（事件簿5）。
- **使った場面**: 診断ビルドで、認識器に渡る音の塊（chunk）の RMS を ~1秒ごとにシリアルログへ出した。

### 3.2 事件簿1: フォント差し替えの罠（やる前に潰した二重の落とし穴）

- **症状（の予感）**: 当初計画は「ステータス文字を出すには日本語フォントを basic → common に差し替える必要がある」だった。
- **調査**: 本当に必要か実機の assets を調べたら、**実行時のテキストフォントは既に**
  `font_puhui_common_20_4.bin`（日本語グリフ入り）だった。`basic` は app に焼く小さな予備にすぎない。
- **原因（もし差し替えていたら）**: ① app バイナリが約 +2MB 肥大してパーティション（3.94MB 枠）を確実に溢れる。
  ② ビルドスクリプトが basic を要求する作りなので、common が assets から**外れて逆に日本語が壊れる**。
- **修正**: 何もしない。**ステータスラベルは画面のフォントを継承するだけで日本語が出る**。
- **教訓**:

> **教訓**: 「Aするには先にBが要る」という前提は、着手前に**実機の事実で裏を取る**。
> このケースは「やらないのが正解」で、計画どおり進めていたら 2 つの壊れ方を同時に踏んでいた。
> 一番安いバグは「書く前に気づいたバグ」。

### 3.3 事件簿2: 顔ステータスと緑LEDが「出ない」（avatar が label を覆う）

- **症状**: 実機で聞き取りを始めても「きいてるよ」が顔に出ない。緑LEDもタップ時しか光らない。
- **調査**: gateway は正しくステータスを送っていた（テスト済み・無罪）。firmware 側の描画を追うと、
  顔（avatar 画像）が瞬き・口パクのたびに最前面化され、その上に重なるラベルを覆っていた。
  緑LEDは「タップ release」だけにハードコードされ、状態連動の点灯経路が死んでいた。
- **原因**: 2.2 で説明した「重なり順」と「点灯経路の不統一」。
- **修正**: 顔を前面化した直後に `PromoteOverlaysLocked()` でラベル類を再昇格。緑LEDは
  「聞き取りに入った瞬間（listening 突入エッジ）」に点灯を移して、タップ/ウェイクワード/ボタンの
  **3経路を1か所に統一**。
- **教訓**:

> **教訓**: 「gateway も firmware も単体では正しいのに、繋ぐと出ない」ときは、
> **両者の境界ではなく、表示の最終段（描画の重なり順）**を疑う。データは届いていた。

### 3.4 事件簿3: 顔ステータスの「ワンテンポ遅れ」（gateway が録音終了まで送らない）

- **症状**: 「きいてるよ」が出るには出るが、**話し終わってから**遅れて出る。聞き始めに出てほしい。
- **調査**: 「きいてるよ」を送っていたのは `hermes_bridge.py`。だがそこに到達するのは
  「タップ→録音終了→音声を HTTP で受信→デコード」が**全部終わった後**。つまり構造的に遅れる。
  firmware の描画（事件簿2で直した側）は無罪だった。
- **原因**: 送るタイミングが「録音が終わってから」だった。
- **修正**: 録音が**始まった最速の瞬間**に送る経路を新設。デバイスが「録音開始（state=="start"）」を
  知らせた直後に `on_listen_started` フックを発火させ、その場で「きいてるよ」を送る:

```python
# esp32_client.py — 録音が開いた瞬間にフックを撃つ
if state == "start" and not is_recording():
    start_recording(session_id)
    if self.on_listen_started is not None:
        asyncio.create_task(self._run_listen_started_hook())

# gateway.py — フックの中身：その場で「きいてるよ」
async def _on_listen_started(self):
    await control.set_device_status_text(self, control.STATUS_LISTENING)
```

`hermes_bridge.py` 側の従来送出は**冪等**（同じ表示を二度送っても無害）なので残置。
これで全体の遷移はこうなりました:

```
録音が開く        →「きいてるよ」  ← ★今回前倒し（タップと同時に出る）
STT 完了          →「考え中」
Hermes が web_search →「調べ中」   （voice_turn_active フラグでゲート）
TTS 完了 / エラー  → すべて消去     （finally で status/字幕/バッジ/LED を必ず戻す）
```

```python
# hermes_bridge.py finally — 終わったら必ず全部リセット
finally:
    gateway.voice_turn_active = False
    await control.set_device_status_text(gateway, control.STATUS_CLEAR)  # ""
    await control.set_device_subtitle(gateway, "")
    await control.set_device_route_badge(gateway, "")
    await control.set_device_led_indicator(gateway, 0, 0, 0)
```

- **教訓**:

> **教訓**: 「出るのが遅い」は描画速度ではなく**送る位置**の問題のことが多い。
> イベントの**いちばん早い合図**（ここでは「録音開始」）まで遡って、そこに通知を貼る。

### 3.5 事件簿4: ②ウェイクワード — 「prob では測れないもの」を RMS で測る

これがフォローで一番学びの大きい一件です。

- **症状**: 「スタックちゃん」と呼んでも検知ゼロ。前セッションは「アイドル時にマイク音声が
  検出器に届いていないのでは？（入力ゼロ説）」まで絞っていた。
- **調査の壁**: 既存の `[DIAG] prob`（確信度）ログでは確定できなかった。MultiNet は時間切れ（TIMEOUT）時に
  結果が空になり、**prob は入力に関係なく常に 0.000** を返す。つまり「prob=0.000」は
  「入力ゼロ」とは限らず、診断として無力だった。
- **決め手**: 「認識器の出力（prob）」ではなく「**入力の物理量（RMS）**」を測ることにした。
  認識器に渡る音の塊そのものの RMS を、`input_enabled`（マイク有効か）と並べてログした。

| 状態 | peakRMS | input_enabled | 認識器の処理 | 検知 |
|---|---|---|---|---|
| 静音 | 10〜20 | 1 | 毎秒31チャンク処理 | — |
| 発話 @ gain 12dB | 約110（-49dBFS） | 1 | 同上 | ゼロ |
| 発話 @ gain 30dB | 608〜1164（-29dBFS） | 1 | 同上 | **ゼロ** |

- **原因（確定）**: 入力ゼロ説は**否定**。マイクは有効で取りこぼし無し、音声経路は完全に健全。
  健全な音量・最敏感の閾値でも検知ゼロ・prob 全 0.000 → **「日本語語を中国語ピンインで MultiNet に
  載せる方式」そのものの限界**と確定した。
- **修正（判断）**: 直さない。タップ/背面なで運用でクローズ（4.1 で詳述）。将来は
  microWakeWord（TFLite で日本語を学習できる）が別フェーズの候補。
- **教訓**:

> **教訓**: 診断ログは「**何を測れて、何を測れないか**」を理解してから設計する。
> prob（出力）は今回「測れないもの」だった。RMS（入力の物理量）に切り替えて初めて
> マイク経路と認識器を完全に切り分けられた。
> 「0.000 だから入力ゼロ」と早合点しなかったのが勝因。

### 3.6 事件簿5: マイクを下げすぎていた（過剰補正）

- **症状**: 事件簿4 の測定中、gain 12dB のとき発話の RMS が約 110 しか出ていなかった。
- **調査**: これは -49dBFS で、音声認識（STT）にも痩せすぎのレベル。
  「ウェイクワードの誤検知を減らそう」とマイク感度を下げ続けた結果の**過剰補正**だった。
- **原因**: ウェイクワード対策のつもりで触った値が、関係ない STT 精度まで巻き込んで悪化させていた。
- **修正**: `set_mic_gain 30`（再ビルド不要・即時）で RMS を約 1000（-29dBFS）の健全レベルへ。
  NVS に保存され、以後の STT にも効く。
- **教訓**:

> **教訓**: 一つの不具合（ウェイクワード）を直そうと一方向に振り続けると、
> **別の機能（STT）を巻き添えにする**。共有パラメータ（マイク感度）を触るときは、
> その値が**他に何へ効くか**を先に洗う。数値で測れば「下げすぎ」は一目で見える。

### 3.7 事件簿6: トークンに日本語が混入（値の二重管理が事故を生む）

- **症状**: ダッシュボードの sudo 設定で、status-api 側の合言葉（トークン）を手打ちした際に
  日本語が混ざり、プロキシ認証が通らなくなった。
- **調査**: gateway と status-api で**同じトークンを2か所に手で書いていた**ことが原因。
- **修正**: status-api 側はトークンの値をコピーせず、gateway と同じ秘密ファイルを
  **参照（EnvironmentFile）**させて一元化した。

```ini
# /etc/systemd/system/status-api.service.d/token.conf
[Service]
EnvironmentFile=-/home/kenji/.yorishiro/secrets.env   # 値はコピーせず参照する
```

- **教訓**:

> **教訓**: 同じ秘密を2か所に書くと、いつか食い違う。秘密は**1か所に置いて参照**する。
> systemd の `EnvironmentFile` は「付箋で本体を触らず環境変数を足す」drop-in の定番。

## 4. 設計判断の記録

### 4.1 ウェイクワードを「粘る」から「クローズ」へ

| 案 | 中身 | 評価 |
|---|---|---|
| ピンイン/閾値を変えて粘る | `su ta ke qiang` → 別表記、閾値 20→12 等 | 測定で「方式の限界」と確定。微調整では届かない |
| microWakeWord に乗り換え | TFLite で日本語「スタックちゃん」を学習 | 有望だが学習・統合コスト大。別フェーズ案件 |
| **タップ/背面なで運用でクローズ** | 既存の明示トリガーで運用 | ✅ 両方正常動作・**設計原則①と完全に一致** |

決め手は技術ではなく**設計原則①「夫婦の会話に割り込まない（明示トリガー優先）」**でした。
ウェイクワードは元々「後で検討」の位置づけ。常時マイクで待ち受ける方式は割り込みリスクがあり、
タップ・背面なでという確実な明示トリガーが既にある以上、無理に粘る理由が薄い。
**測定で限界を確定 → 原則に照らしてクローズ**、という順番がきれいでした。

### 4.2 中立姿勢を「再ビルド」から「NVSランタイム + ジョイスティック」へ

ユーザー提案「首の角度のような見た目の好みを 25 分の再ビルドで合わせるのは無駄。
ダッシュボードでライブ調整して保存したい」を採用。設計のポイント:

- **中立を単一の出所に束ねた**: 中立角を見ていた3経路（起動時 / 近接・タッチ復帰 / アイドル復帰）が
  バラバラに定数を参照していたのを、`neutral_yaw_`/`neutral_pitch_`（起動時に NVS から解決、既定 0/38）に
  一本化。「一度保存すれば、どの経路から戻っても同じ新しい中立になる」。
- **ライブと保存を分けた**: ドラッグ中は `set_head_angle`（動かすだけ・揮発）、「保存」ボタンで初めて
  `set_neutral_pose`（NVS 永続）。throttle 120ms（約8Hz）で連投を抑える。
- **起動時の安全シーケンスを壊さない**: 起動時にサーボの現在角と目標角を同じ中立値にして
  `start == target` を保つ（急な振り回しを防ぐ既存の安全条件）。

### 4.3 ダッシュボードを git 管理外（`~/razer-dashboard/`）に置く

- ダッシュボード（`status_api.py` / `dashboard.html`）は本リポジトリに**入れていない**。
- 理由: ① **トークン等の秘密**を扱う受付なので、誤コミットの事故面を物理的に無くす。
  ② razer-server 固有の運用物で、fork として upstream に出すものでもない。
- 代償: バージョン管理が手動バックアップ（`*.bak-YYYYMMDD`）頼みになる。今は「ファイル編集が即デプロイ」の
  手軽さを優先。将来 git 化するなら、秘密を抜いた状態で別リポジトリにするのが筋。

### 4.4 やらなかったこと（意図的に）

- **フォント差し替え**: 事件簿1のとおり不要かつ有害。やらないのが正解だった。
- **ウェイクワードの深追い**: 4.1 のとおり、方式の限界を測定で確定した時点で打ち切り。
  「動くまで粘る」より「原則に照らして区切る」を選んだ。
- **マイク感度の自動調整（AGC 等）**: 今回は手動スライダー + 数値計測で十分。
  自動化は誤動作時の説明が難しくなるので、まず手で納得してから。
