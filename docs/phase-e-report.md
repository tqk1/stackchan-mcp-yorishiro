## 1. 全体像

### 1.1 何を作ったか（一言で）

Phase D で「30分に1回、無音の仕草をする」ようになった StackChan に、
**「伝える価値がある情報があるときだけ、一言話す」** 能力を足しました。

達成した体験（2026-06-12 朝、実機ログで確認済み）:

```
（朝 7:53、誰も話しかけていないのに）
StackChan:「今日は雨が降りそうだよ、降水確率20%。傘を忘れずにね」 ← 気象庁の実データ
（次の tick、7:54）
StackChan:「今日のメモに『牛乳を買う』ってあるよ」                ← 今朝書いたメモ
（その次の tick、7:55）
StackChan:（言うことがもう無いので黙って表情だけ変える）          ← 沈黙がデフォルト
```

### 1.2 一番大きな学び — 「何を作るか」は計画段階で覆る

Phase D のレポートの宿題には「heartbeat 第2段階 = Hermes に文脈を渡して短い一言を生成」と
書いてありました。つまり当初の構想は **ランダム雑談型** でした。

ところが計画段階の相談で、ケンジさんの一言で方向が変わります:

> 「heartbeatによって無意味な一言発話や動きをして欲しいわけではありません」

ここから **通知型**（沈黙がデフォルト、言うべきことがあるときだけ話す）にコンセプトを転換しました。
結果として:

- LLM を一切使わない、決定的でテスト可能な設計になった（テスト 42 件で全分岐を検証）
- 「空振り発話をどう抑えるか」という難問が、構造的に**発生しなくなった**
  （雑談型は「何か言う」が前提なので抑制が難しい。通知型は「言うことが無ければ黙る」が自然）

> **教訓**: 宿題リストの文言をそのまま実装しない。「そもそも何のための機能か」を
> 計画段階でユーザーと擦り合わせると、実装が簡単になる方向に問題が変わることがある。

## 2. 採用した技術の解説

### 2.1 気象庁 bosai API — 「役所が無料で配っている天気の生データ」

- **何か（一言）**: 気象庁が `www.jma.go.jp/bosai/` 配下で配信している防災情報の JSON フィード。
  API キー・登録・課金が一切不要
- **なぜ必要か**: 「雨が降りそうなときだけ教える」には降水確率と警報・注意報の生データが要る。
  民間天気 API はキー管理や無料枠の制約があるが、これは公式かつ無条件
- **喩え**: 市役所の掲示板。誰でも見られるし、内容は公式。ただし「掲示板の読み方マニュアル」は
  配られていない（非公式 API なので構造のドキュメントが無い）
- **使った場面**: `gateway/stackchan_mcp/weather.py`。使う URL は 3 つだけ:

```
bosai/common/const/area.json                 ← エリアコード表（守口市 = 2720900 を特定）
bosai/warning/data/warning/270000.json       ← 大阪府の警報・注意報（発表中/解除の別付き）
bosai/forecast/data/forecast/270000.json     ← 大阪府の予報（降水確率 pops を含む）
```

ハマりどころを2つ実データで確認して処理に入れてあります:
1. **「解除」も載っている** — 解除済みの注意報がフィードに残るので、status が
   「発表」「継続」のものだけ拾う（今朝の実データに `乾燥注意報 status=解除` が居た）
2. **過ぎた時間帯の降水確率は空文字** — `pops: ["", "30", ...]` のように歯抜けになるので
   `int()` できないものは読み飛ばす

### 2.2 checker パターン — 「情報源ごとの当番さん」

- **何か（一言）**: 「いつ確認するか（ウィンドウ）＋ 何を確認するか（検出）＋ どう言うか（文生成）」を
  1セットにした小さなユニット。`_check_weather()` と `_check_memo()` の2つで開始
- **なぜ必要か**: 将来 SwitchBot 人感センサーや CO2 センサーを買ったとき、
  「CO2 が高いよ」当番を **1メソッド追加するだけ** で組み込めるようにするため
- **喩え**: 旅館の仲居さん。朝の担当は天気を見て必要なときだけ傘を勧め、夕方の担当は
  忘れ物がないか見て必要なときだけ声をかける。**用が無ければ現れない**のがポイント
- **使った場面**: `heartbeat.py` の `_tick_speak()` が当番を順に呼ぶ。優先度は天気 → メモ。
  1 tick で話すのは最大 1 件

### 2.3 多層抑制（defense in depth）— 「五重の戸締まり」

設計原則1「夫婦の会話に割り込まない」を、**1つの仕組みに頼らず5つの独立した層**で守ります。

| 層 | 何を守るか | 実装 |
|---|---|---|
| ① クワイエットアワー | 夜間・早朝（22:00-06:30） | Phase D の `is_quiet()` 流用 |
| ② tts_lock | StackChan 自身が話し中・聞き取り中 | Phase D の既存ガード |
| ③ 録音スロット | ユーザーがタップして録音している最中 | `audio_stream.is_recording()` |
| ④ 対話クールダウン | **会話の余韻**（直近20分の対話） | ★今回新設（下記） |
| ⑤ 日次上限 | 暴走時の安全弁（デフォルト3回/日） | ★カウンタ + state 永続化 |

④が今回の設計の急所です。会話の流れはこうなっています:

```
タップ → 録音 → STT → [Hermes が考える: 最大120秒] → 返事の TTS
                        ~~~~~~~~~~~~~~~~~~~~~~~~~~
                        この間、②も③も「空き」になる！
```

つまり「話しかけた直後、StackChan が返事を考えている最中」は既存ガードの死角で、
そこに通知が割り込む事故が起こり得ました。対策として、タイムスタンプを
**voice_turn 処理の冒頭**（録音が届いた瞬間）で記録します。
これで考え中の120秒もクールダウン圏内に入ります。

> **教訓**: 「鍵をかけたか」より「どの瞬間に鍵が開いているか」を探す。
> ロック系の設計はロックが**外れている時間帯**を時系列で書き出すと穴が見える。

### 2.4 state ファイル — 「電源を切っても消えない出席簿」

- **何か（一言）**: `~/.stackchan/heartbeat_state.json`。「今日天気を言ったか」「どのメモを
  もう読み上げたか」「今日何回話したか」を記録する小さな JSON
- **なぜ必要か**: gateway は systemd 管理で再起動が日常的にある。記憶がプロセス内だけだと、
  再起動のたびに同じメモを読み上げる「壊れたレコード」になる
- **喩え**: 黒板（プロセスのメモリ）ではなくノート（ファイル）に書く。停電しても残る
- **使った場面**: 今朝の実機検証で実際に生成された中身:

```json
{"weather_done": "2026-06-12", "speak_count_date": "2026-06-12",
 "speak_count": 2, "reminded": {"メモ.md": 1781212766}, "memo_done": "2026-06-12"}
```

`reminded` の値はメモの **mtime（最終更新時刻）**。同名メモでも内容を書き足せば
mtime が変わるので「更新されたらまた教えてくれる」が自然に成立します。

### 2.5 pure 関数分離 — 「計りは台所、味見は別室」

- **何か（一言）**: ネットワーク取得（`check_weather`）と判定（`judge_weather`）を別関数に分けること。
  判定関数は「JSON を渡すと発話文 or None を返す」だけで、外部に一切触れない（= pure）
- **なぜ必要か**: 気象庁 API は非公式でドキュメントが無い。構造が変わったときに備えて
  判定ロジックを**ネット無しで網羅テスト**できる形にしておく（警報あり/降水確率ちょうど閾値/平常/壊れた JSON…全11ケース）
- **喩え**: レシピの「計量」と「加熱」を分ける。計量だけなら火を使わず何度でも練習できる
- **使った場面**: `tests/test_weather.py`。実 API は別途1回だけ手で叩いて、フィクスチャ（テスト用の
  偽 JSON）が実物と同じ形であることを確認した

## 4. 設計判断の記録

### 4.1 一言の生成元 — なぜ LLM を使わなかったか

| 案 | 速度 | コスト | 制御性 | 採用 |
|---|---|---|---|---|
| Hermes（当初の宿題案） | 遅い | 外部トークン恒常消費 | プロンプト頼み。自発 tick からツールが走るリスク | |
| ローカル LFM2.5 | 速い | 無料 | 出力の揺らぎは残る | |
| **テンプレート固定** | 即時 | ゼロ | 完全に決定的・テスト可能 | ✅ |

通知型では「何を言うか」の中身（降水確率20%、メモの内容）はデータがそのまま運んでくれるので、
LLM が文章を考える余地がほぼ無い。**情報が主役なら定型文で足りる**。
「言い回しだけ LLM に磨かせる」拡張点はコードコメントに残してあります。

### 4.2 ウィンドウ方式 — なぜ「毎 tick チェック」にしなかったか

天気は朝（06:30-09:30）に1回、メモは夕方（18:00-21:00）に1回だけ。理由:

- 通知には**適時性**がある。傘の話は出がけに、買い物メモは買い物に行く時間帯に
- 1日中チェックすると「言う口実」が増えてうるさくなる方向に倒れる。
  本プロジェクトは常に**静かな方向に倒す**（設計原則1）

### 4.3 やらなかったこと（意図的に）

- **SwitchBot 消し忘れ検知**: 「意図的な点灯」と「消し忘れ」の区別ルールが難しく、
  誤通知=信頼低下のリスクが高い。センサー購入後の文脈（人感）と合わせて再検討
- **天気の「いい知らせ」**: 「今日は晴れだよ」は言わない。情報価値が低く、
  毎日話す=ノイズになる。沈黙がデフォルトの原則を守る
- **Hermes への状況報告**: heartbeat が話した内容を Hermes の会話履歴に注入することも
  できるが、会話の文脈を汚すので見送り（Phase D4 の「経路分離」と同じ思想）

## 5. 用語集

| 用語 | 一言で | 喩え |
|---|---|---|
| 通知型 heartbeat | 価値ある情報があるときだけ自発発話 | 用があるときだけ声をかける仲居さん |
| checker | 情報源1つ分の「確認＋判定＋文生成」ユニット | 持ち場ごとの当番 |
| bosai API | 気象庁の無料 JSON フィード | 市役所の掲示板 |
| class20 / office コード | 気象庁の地域 ID（市区町村/都道府県） | 郵便番号の気象庁版 |
| pops | 降水確率の配列（時間帯別） | 天気予報の「降水確率 20%」の生データ |
| pure 関数 | 外部に触れず入力→出力だけの関数 | 火を使わない計量作業 |
| state ファイル | プロセス外に置く記憶 | 黒板でなくノートに書く |
| クールダウン | 直近対話からの沈黙時間 | 会話の余韻を邪魔しない間合い |
| monotonic clock | 巻き戻らない経過時間計 | ストップウォッチ（壁時計と違い時刻合わせの影響を受けない） |
| 二重 opt-in | 2つの環境変数が揃って初めて有効 | 金庫の2本鍵 |
| drop-in (systemd) | サービス設定の差分ファイル | 本体を書き換えずに付箋で上書き |
