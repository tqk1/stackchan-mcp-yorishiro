## 3. 何をやったか（時系列）

### 3-1. C-fix: Hermes→gateway MCP 接続の修復（回帰）

朝の作業で gateway の `/mcp` に STACKCHAN_TOKEN の Bearer 認証を有効化した結果、
**Hermes 側の設定にヘッダーが無く 401 → 5 回リトライ → giving up** していた
（ログのバックオフ間隔 2s→4s→8s→16s→32s が発見の手がかり）。

修正:
1. `~/.hermes/config.yaml` の `mcp_servers.stackchan` に `headers: {Authorization: "Bearer ${STACKCHAN_TOKEN}"}` を追加
   — Hermes は設定値の `${VAR}` を環境変数から展開できる（`mcp_tool.py:2234`）
2. トークン値は `~/.hermes/.env` へ（ユーザーが `grep ... >> ...` で転記。Claude は秘匿ファイル不可読の運用）
3. hermes-gateway 再起動 → `/mcp` 200 OK

**学び**: トークン認証を片側に入れたら、その口を使う**全クライアント**（ESP32 だけでなく Hermes も）の設定を同時に更新する。

### 3-2. C2: SwitchBot 家電操作

- 事前調査: Home Assistant はどこにも無い → ゼロから立てるより **SwitchBot API v1.1 直結**が現実的と判断
- 前提確認: Hub Mini ×2（リビング・寝室）+ 赤外線リモコン 5 台がアプリ登録済み。開発者トークンを発行して `~/.yorishiro/secrets.env` へ
- 実装: `stackchan_mcp/switchbot.py`（fork 専用、`hermes_bridge.py` と同じ流儀）
  - 認証: `sign = base64(HMAC-SHA256(secret, token + t + nonce))`（公式 README の Python 例準拠）
  - MCP ツール 3 つ: `switchbot_list_devices` / `switchbot_get_status` / `switchbot_send_command`
  - **既存の :8767 MCP HTTP に同居** — 新サービス・新ポート・Hermes 再登録すべて不要
  - ESP32 を使わないツールなので、デバイス直列キュー（single-flight queue）を**バイパス**
- E2E: 音声「SwitchBotに登録されているデバイスを教えて」→ Hermes がツールを呼んで実デバイスを音声で報告 → 「リビングの電気をつけて」→ **実灯確認** ✅

### 3-3. C3: 応答ルーティング層

- モデル選定（実測ベンチ）: **LFM2.5-1.2B-JP Q4_K_M**（warm 0.48〜0.50s、0.7GB）。gemma3:4b（1.45〜2.03s）より速く、VRAM 6GB 制約でも whisper と余裕で同居
- `local_llm.py` の `decide_route()` は**純関数のルールベース**:
  1. Hermes マーカー語（調べ/検索/天気/予定/覚えて/なぜ/説明して…）→ Hermes
  2. **家電操作語（つけて/消して/電気/エアコン/テレビ…）→ Hermes** ← E2E 前に発見した穴。ローカルモデルはツールを呼べないので、短い命令文がローカルに吸われると家電が動かない
  3. NFKC 正規化 30 文字以下 → ローカル
  4. それ以外 → Hermes
- opt-in 設計: `STACKCHAN_LOCAL_LLM_MODEL` 未設定なら従来挙動と完全同一。ローカル失敗時は Hermes へフォールバック（会話は死なない）
- デプロイ: systemd drop-in（`docs/deploy/stackchan-gateway.service.d/local-llm.conf`）。外せばロールバック
- 小型モデルは「今日」を知らないので、システムプロンプトに日付を注入（「今日は何曜日?」対策）

### 3-4. C1: LTR-553 近接リフレックス → 実機検証で「シェルが塞いでいる」と判明

- `i2c_scan`（外部 Grove バス）は空 → 「近接センサー無し」と一度は結論しかけたが、
  **ユーザーの指摘で CoreS3 内蔵の LTR-553ALS-WA（内部 I2C 0x23、近接+照度）を発見**。
  firmware が単にドライバを持っていなかっただけ
- Si12T タッチポーリングと同じ esp_timer パターンでドライバ + リフレックスを実装し、
  flash して実機キャリブレーションを実施。結果:
  - チップは生きている（part_id=0x92 応答、I2C 読み取り正常）
  - **感度最大（gain x64・LED 15 パルス・100mA）でも ps_raw は ~380 で固定**。
    手を 1cm まで近づけてもゼロ変化 → **前面シェルに LTR-553 用の開口がなく、
    外界の IR が物理的に届かない**と結論
  - ~380 という値は前面パネル内面の反射（クロストーク）だけを見ている
- 副所見: 初版の閾値 200 はベースライン 380 より低く、**毎起動時に誤発火**していた
  （起動 5 秒後に首が上を向く）。`PROX_REFLEX_ENABLED=false` + 閾値 700 に修正して再 flash
- ドライバと `get_touch_state` の ps_raw 診断は温存（シェル加工や将来の流用に備える）
- 本来の目標「近づくと向く」(1〜2m) は **M5Stack ToF Unit (VL53L0X、Grove、レーザー測距 2m)**
  で実現する方針。購入はユーザー判断待ち

**デバッグの流れ（学び）**: ps_raw=0 → 「設定か物理か」を切り分けるため
(1) 感度を仕様上限へ (2) 飽和フラグを可視化 (3) 2 秒ごとの生値ログ、の 3 点セットで再 flash →
「ベースラインは出る（=IC は動く）が手に無反応（=光路がない）」で物理閉塞と確定。
**生値の連続ログがあれば 1 往復で結論が出る** — 最初からデバッグ可視性を仕込む価値がある。

## 4. 用語解説（今日出てきたもの）

| 用語 | 説明 |
|---|---|
| **Bearer トークン認証** | HTTP ヘッダー `Authorization: Bearer <秘密文字列>` で身元確認する方式。gateway の /mcp と WS の両方がこれで保護されている |
| **指数バックオフ** | 失敗時のリトライ間隔を 2s→4s→8s… と倍々にする戦略。ログでこのパターンを見たら「クライアントの自動再接続が失敗し続けている」サイン |
| **HMAC-SHA256 署名** | 秘密鍵でメッセージの改ざん検知コードを作る方式。SwitchBot API はリクエストごとに `token+timestamp+nonce` を署名させ、盗聴リプレイを防ぐ |
| **nonce** | 一度きりの使い捨て乱数。同じリクエストの再送（リプレイ攻撃）を無効化する |
| **single-flight queue** | ESP32 への命令を 1 つずつ直列実行する gateway の仕組み（サーボが同時命令で暴れないように）。SwitchBot ツールは ESP32 と無関係なのでバイパス |
| **opt-in 設計** | 環境変数を設定した時だけ新機能が有効になる方式。未設定なら従来と完全同一なので、ロールバックが「設定を消すだけ」になる |
| **keep_alive (Ollama)** | 推論後もモデルを VRAM に常駐させる時間。これが無いと毎回ロードし直し（コールドスタート ~3.4s）になる |
| **NFKC 正規化** | 全角/半角・互換文字を統一する Unicode 正規化。「ﾆｭｰｽ」と「ニュース」を同じ文字列として扱える |
| **LTR-553ALS-WA** | CoreS3 内蔵の照度+近接センサー。スマホの「耳に当てたら画面オフ」と同じ原理（IR LED の反射量で近接判定） |
| **PS / ALS** | Proximity Sensor（近接）/ Ambient Light Sensor（照度）。LTR-553 は両方入り |
