## 1. 全体像

### 1.1 何をしたか（一言で）

買ってきたままの StackChan（M5Stack 公式キット）を、**自分のプログラムで動く器に
作り変えました**。具体的には:

1. firmware（マイコンの中で動くプログラム）をビルドする環境を PC に構築
2. 公式 firmware を消して、この fork の firmware に書き換え
3. 「触ると首が動く」「頭をなでると反応する」を実機で確認
4. 自宅 WiFi に接続し、PC 上の gateway と通信できるようにした
5. 顔（アバター画像）を自作して画面に表示

### 1.2 Before / After

```
【Before】                          【After】
 M5Stack 公式 firmware              この fork の firmware
 ・M5Stack 社のクラウドと通信        ・自宅の razer-server とだけ通信
 ・中身はブラックボックス            ・全コードが手元にあり改造可能
 ・顔は公式のもの                    ・顔は自作 PNG 14 枚（表情差し替え可）
                                    ・PC から MCP で 30 個のツールを操作可能
```

### 1.3 開発の流れ（全体地図）

```
[PC 側の準備]                  [実機側の作業]
 Docker で ESP-IDF 環境  ───►  USB ケーブルで接続
        │                        │
        ▼                        ▼
 firmware をビルド       ───►  esptool で書き込み (flash)
 (xiaozhi.bin 生成)              │
                                 ▼
                          シリアルログで起動確認
                          「首が動く」「なで反応」を目視
                                 │
                                 ▼
                          WiFi 設定投入（配網モード）
                                 │
                                 ▼
                          mDNS で gateway を自動発見
                          → WebSocket 接続 → 顔表示 ✅
```

## 2. 採用した技術の解説

### 2.1 ESP32-S3 / M5Stack CoreS3 — 「小さな全部入りコンピュータ」

- **何か**: StackChan の頭脳になっているマイコン（マイクロコントローラ）。
  CPU・メモリ・WiFi・Bluetooth が1チップに収まっていて、消費電力は PC の数百分の一。
- **PC との違い**: OS（Windows や Linux）が**載っていません**。プログラムは1個だけが
  常に全力で走り続けます。だから「アプリをインストール」ではなく「firmware を書き込む」
  という言い方をします。
- **CoreS3 の構成**: ESP32-S3 + 2インチタッチ画面 + カメラ + マイク + スピーカー +
  各種センサー。StackChan キットはこれに首サーボ2個を足したものです。

### 2.2 firmware — 「機械に焼き込む唯一のプログラム」

- **何か**: マイコンの中で動くプログラム本体。電源を入れると即座にこれが走り始めます。
- **喩え**: PC のプログラムが「OS の上で動くアプリ」だとすると、firmware は
  「機械そのものの人格」。書き換える＝人格を入れ替える、です。
- **今回の出自**: `78/xiaozhi-esp32`（中国発の AI スピーカー firmware）を
  `kisaragi-mochi/stackchan-mcp` が StackChan 用に改造し、それを本プロジェクトが
  fork（コピーして独自進化）したもの。**三段重ねの fork** です。

### 2.3 ESP-IDF — 「ESP32 専用の開発キット」

- **何か**: espressif 社（ESP32 のメーカー）公式の開発環境。C/C++ のコードを
  ESP32 で動くバイナリに変換（ビルド）するためのツール一式です。
- **なぜ必要か**: PC 用のコンパイラでは ESP32 用のプログラムは作れません。
  CPU の種類が違うため、専用の「翻訳機」が要ります（クロスコンパイルと言います）。

### 2.4 Docker — 「開発環境を箱ごと借りる」

- **何か**: プログラムを「環境ごとパッケージ化した箱（コンテナ）」で動かす仕組み。
- **なぜ使ったか**: ESP-IDF を PC に直接インストールすると、特定バージョンの
  Python やツール群が大量に入り、他の作業と衝突しがちです。Docker なら
  `espressif/idf:v5.5.2` という**完成済みの箱**を借りてきて、その中でビルドだけ
  実行し、PC 本体は汚れません。
- **喩え**: 自宅の台所を改造する代わりに、フル装備のキッチンカーを呼ぶ。

```bash
# 実際に使ったコマンド（概念）
docker run --rm -v "$PWD":/project -w /project espressif/idf:v5.5.2 \
  python ./scripts/release.py stackchan
# → build/xiaozhi.bin（書き込み用バイナリ）ができる
```

### 2.5 esptool と flash — 「マイコンへの書き込み」

- **何か**: ビルドしたバイナリを USB 経由で ESP32 の内蔵ストレージ（フラッシュメモリ）
  に書き込むツール。書き込む行為自体も「flash する」と呼びます。
- **重要な学び — パーティション**: ESP32 のストレージは区画割りされています。
  - **app 領域**: プログラム本体。ここだけ書き換えれば更新できる
  - **NVS 領域**: WiFi のパスワードなどの設定保存場所（電源を切っても消えない手帳）
  
  初回は全領域を書きましたが、以降は **app だけ書き換える**ことで
  「WiFi 設定を保持したままプログラムだけ更新」ができました。これを知らないと
  更新のたびに WiFi 設定をやり直す羽目になります。

### 2.6 シリアル通信 — 「マイコンの独り言を聞く窓」

- **何か**: USB ケーブル越しにマイコンのログ（printf の出力）を読む通信路。
  画面のないマイコン開発では、これが唯一のデバッグ手段です。
- **ハマりポイント（重要）**: CoreS3 の USB-Serial/JTAG は、
  **ポートを開くだけでデバイスがリセット（再起動）されます**。
  「動作確認しようとログを覗いた瞬間に再起動して状況が変わる」という
  観測者効果のような罠で、WebSocket 接続の検証中は触らないルールにしました。

### 2.7 sdkconfig / Kconfig — 「ビルド時の仕様書」

- **何か**: firmware のビルド設定ファイル。「mDNS を使うか」「接続先 URL は何か」
  などのスイッチが数千行並んでいます。
- **ハマりポイント**: このファイルはビルドスクリプトが**毎回自動再生成する**ので、
  直接手で書き換えても消えます。正しい上書き手段は `sdkconfig.defaults.local`
  という別ファイル（git 管理外）に差分だけ書く方法でした。
  「自動生成物は直接編集しない。入力側を変える」は開発全般の鉄則です。

### 2.8 mDNS — 「LAN 内の名前で呼び出す電話帳」

- **何か**: 「このネットワークに gateway いますか？」と LAN 内に呼びかけて、
  IP アドレスを知らなくても相手を見つけられる仕組み（multicast DNS）。
- **なぜ必要か**: 当初 StackChan は gateway の場所（`ws://192.168.0.19:8765/`）を
  手入力する必要がありました。mDNS を有効にすると**自動発見**になり、
  IP が変わっても追従できます。

## 3. 工程の流れ（時系列）

### 工程1: ビルド環境構築（T1〜T5）

| 作業 | 内容 | 結果 |
|---|---|---|
| Docker 確認 | 既にインストール済みと判明 | 流用 |
| esptool 導入 | 書き込みツールを専用 venv に | `~/.venvs/esptool/` |
| submodule 取得 | firmware が依存する外部ライブラリ | smooth_ui_toolkit |
| ESP-IDF 取得 | Docker イメージを pull | espressif/idf:v5.5.2 |

### 工程2: ビルドと初回 flash（T6〜T8）

`release.py stackchan` で 9.6MB のバイナリが完成。USB 接続した CoreS3
（`/dev/ttyACM0` として認識）に 460800bps で書き込み、39秒で完了。
`Hash of data verified.` の表示 = 書き込んだデータが壊れていないことの検証合格です。

### 工程3: 起動確認（T9〜T10）

シリアルログで以下を確認:

```
Servo power ENABLED via PY32 pin 0   ← サーボに電源が入った
Boot pre-init ReadPos 626→759        ← 首が物理的に動いた（数値でも裏付け）
Si12T: init OK                       ← 背面タッチセンサー初期化成功
touch event: STROKE duration=1699ms  ← 「なでた」ことを検知
```

**ユーザー目視でも「首が動く」「なでると反応する」を確認** — Phase A の
完了条件「触ると首が動く」を達成しました。

### 工程4: WiFi 投入（B0-4）— 配網モードという仕組み

WiFi 未設定の firmware は、自分自身が WiFi アクセスポイント（`Xiaozhi-E79D`）になって
待ち受けます。スマホでそれに繋いで `http://192.168.4.1` を開くと設定画面が出る —
**家電の WiFi 設定でおなじみのあの方式**（配網モード / captive portal）です。
SSID とパスワードを入れると、以降は自宅 WiFi に自分から繋ぎに行きます。

### 事件簿1: 「接続先が分からない」→ mDNS が無効だった

WiFi には繋がったのに、gateway への接続先 URL が空で迷子になりました。
調べると **mDNS（自動発見）の設定がビルド時に無効になっていた**ことが原因。

さらに深掘りすると、無効になった理由は設定の生成順序でした:
「ボード選択前に設定が確定 → 後からボード固有の設定を足しても反映されない」。
解決策は前述の `sdkconfig.defaults.local` に mDNS 有効化と予備 URL を書いて
リビルド。以降、StackChan は起動すると自動で gateway を見つけて繋がります。

### 事件簿2: 顔の画像がどこにも売っていない

「アバターは公式（m5stack-avatar）を移植」という当初計画は、調査で
**そもそも不要**と判明しました（fork firmware に表示機構が実装済み）。
ただし**顔の画像データそのものは存在しない** — m5stack-avatar は
「プログラムで顔を描くライブラリ」であって、PNG 画像は配布されていなかったのです。

そこで Python の画像ライブラリ PIL で**公式風の顔を自作**しました。
idle / happy / angry など 14 表情を1つのスクリプト（`scratch/gen_avatar_faces.py`）で
生成し、firmware に焼き込み。**実機の画面に顔が表示されることを確認**して
Phase B-0 完了です。

> **教訓**: スクリプトで生成したので、顔のデザイン調整は「数値を変えて再実行」
> するだけ。手描きと違い、何度でもやり直せる資産になりました。

## 4. 設計判断の記録

### 4.1 ビルド環境: Docker か、直接インストールか

| 観点 | Docker ✅ | 直接インストール |
|---|---|---|
| PC 環境の汚れ | なし | Python 等が大量に入る |
| バージョン固定 | イメージ名で厳密に固定 | 更新で壊れるリスク |
| 実行速度 | わずかに遅い | 速い |

ビルドは1日数回しかしないので、速度より**再現性と清潔さ**を取りました。

### 4.2 アバター入手の3経路

| 経路 | 内容 | 判断 |
|---|---|---|
| (a) ホットスワップ | 実行中に画像を差し替え | 揮発性（電源断で消える）。表情の動的変更用に温存 |
| (b) ビルド焼き込み ✅ | 画像を firmware に同梱 | 確実・電源断でも残る。**採用** |
| (c) 既製画像を探す | 配布物を流用 | **存在しなかった**ので不可能と判明 |

「探す → なければ作る → 作るならスクリプトで」の順で進めたのがポイントです。

### 4.3 接続先の解決: 優先順位の多層化

gateway の場所を見つける手段を3段重ねにしました:

```
1. NVS の保存値（ユーザーが明示設定した場合は最優先）
2. mDNS 自動発見（通常はこれ）
3. ビルド時の予備 URL（mDNS が死んでいた時の最後の砦）
```

Phase B の録音停止（VAD + 再タップ + 30秒上限）と同じ思想 —
**「賢い機能 + 素朴な保険」の多層構造**です。

### 4.4 やらなかったこと

- **m5stack-avatar ライブラリの移植**: 調査の結果不要と判明。
  「計画に書いてあっても、着手前に前提を再確認する」が効いた例です。
- **firmware のゼロからの自作**: 三段 fork の上に乗ることで、
  音声・表示・センサーの実装をすべて流用できました。
